# GA4 Web 用户增长与转化分析

## 项目概述

基于 Google Analytics 4（GA4）公开电商演示数据，完成从事件级字段审计、数据清洗到漏斗、留存、渠道和用户分层分析的完整链路，并用 MySQL 复现核心指标、以 Tableau 呈现结果。

本项目对应互联网产品数据、用户增长、数据运营和 BI 场景。数据来自公开数据集，不将分析结论表述为真实公司收益。

## 业务问题

1. 用户在商品浏览、加购、结账和购买之间的转化表现如何？
2. 用户首次活跃后的留存趋势如何？
3. 不同首次获客渠道的用户质量有何差异？
4. 如何依据活跃天数、订单与收入构建用户分层，为后续运营动作提供观察切口？

## 数据与处理边界

- 数据源：`bigquery-public-data.ga4_obfuscated_sample_ecommerce.events_*`。
- 分析窗口：2020-11-01 至 2021-01-31，共 92 天。
- 原始审计规模：4,295,584 条事件、270,154 个用户、17 类事件。
- 本地事件级分析表：2,742,274 行；全量用户分析口径为 270,051 个去重用户。
- 原始事件表一行代表一次用户事件；用户、日期、渠道和漏斗步骤指标均需重新聚合，不能直接相加。

## 核心分析口径

- **步骤到达漏斗**：`view_item → add_to_cart → begin_checkout → add_payment_info → purchase`，统计到达各步骤的去重用户数；不强制校验严格时间顺序，因此不等同于严格路径漏斗。
- **首次活跃 cohort 留存**：以分析窗口内首次活跃日期分组；Day N 留存 = 当 cohort 第 N 天仍有任意事件的去重用户数 / cohort 用户数，不是注册留存。
- **渠道**：使用 GA4 `traffic_source` 的首次来源字段比较首次获客质量，不等同于每次 session 的实时来源。
- **订单与收入**：订单在 `purchase` 事件中按有效 `transaction_id` 去重；收入和有效订单 ID 缺失情况分别处理，不能强行视为同一口径。
- **活跃分层**：以活跃天数为主要依据，避免将单次访问中产生的多条事件误读为高活跃。

## 分析边界

- 渠道中的 `<Other>`、`(data deleted)` 等值不能过度解读。
- 漏斗流失、渠道差异和留存差异仅用于提出待验证的优化方向，不能据此直接断言因果。
- 公开数据项目中不虚构上线结果、增长收益或真实商业决策影响。
